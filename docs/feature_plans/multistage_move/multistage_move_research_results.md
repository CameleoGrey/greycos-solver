TimeFold’s Multistage Move Mechanism
Purpose and Concept of Multistage Moves
TimeFold’s “multistage move” mechanism refers to using composite or multi-step moves in its search algorithm. Instead of changing only one element at a time, a multistage move can make several coordinated changes in one step. The purpose of this is to allow bigger jumps in the solution space, helping the solver escape local optima or score traps where single small moves aren’t effective
docs.timefold.ai
. In some scenarios, improving the solution requires multiple simultaneous adjustments – if the solver only tries one change at a time, it might see no immediate benefit and get stuck. Multistage moves address this by bundling those adjustments together so that the move as a whole yields a better score. For example, TimeFold suggests adding coarse-grained moves (multi-change moves) alongside fine-grained ones to “directly get out of a score trap with a single move”
docs.timefold.ai
. This way, the solver can overcome situations where a constraint violation can only be fixed by moving several items at once.
Integration in TimeFold’s Local Search Framework
In TimeFold’s local search (the phase after an initial solution is constructed), the solver repeatedly evaluates neighboring solutions generated by moves. Multistage moves are integrated as advanced move types within this neighborhood exploration. The solver’s Move Selector can produce both simple moves (e.g. assigning one entity to a new value) and complex moves that involve multiple changes. In practice, a typical configuration will include basic single-entity moves for completeness, combined (unioned) with more coarse-grained multi-entity moves
docs.timefold.ai
. Including simple moves (like single swaps or reassignments) ensures every possible solution is reachable through some sequence of moves, while the larger moves provide shortcuts to jump to promising regions of the search space
docs.timefold.ai
. All these moves are part of the local search’s neighborhood; the metaheuristic (Tabu Search, Simulated Annealing, etc.) will consider them at each step and decide which move to apply next based on score improvement and acceptance criteria. In essence, the multistage move mechanism augments the local search by broadening the set of neighbors: the solver can try not only one-change neighbors but also multi-change neighbors in each iteration. This improves the solver’s ability to navigate complex solution landscapes and avoid getting stuck on suboptimal plateaus.
Key Strategies and Examples
TimeFold provides several out-of-the-box move types that exemplify the multistage move mechanism. These moves follow key strategies like combining atomic changes into one step and targeting known difficult structures in the problem. Some notable examples and principles include:
Swap Moves: A swap move exchanges the values of two planning entities in one operation. Although it could be seen as two individual “change” moves done back-to-back, doing it as a single atomic move avoids any invalid intermediate state
docs.timefold.ai
. For example, swapping two lectures’ room assignments in one step means the solver never creates a moment where both lectures share a room (a hard conflict) – the swap is evaluated as one neighbor that either fixes or worsens the score directly
docs.timefold.ai
. This illustrates a principle of multistage moves: by grouping changes, the solver can consider neighbor solutions that would be inaccessible or temporarily infeasible via one-at-a-time changes.
Pillar Moves: Pillar moves involve moving a set of entities together. A “pillar” is a group of entities that currently have the same value (e.g. multiple tasks all assigned to the same resource). A pillar change move might reassign an entire group to a new common value in one step, and a pillar swap move can exchange the assignments of two groups. This is useful for structures like workload balancing, where shifting a whole group can improve the score. Since the number of possible subsets of a group is huge (exponential in group size), TimeFold’s move selector handles this in stages. It will select sub-pillars (subsets of the group) using randomized just-in-time selection instead of generating all combinations
docs.timefold.ai
. In other words, the move construction is multi-stage: first pick a random subset of entities, then pick a new target value (or another subset to swap with). This on-the-fly generation keeps the search efficient even though pillar moves are combinatorially complex
docs.timefold.ai
.
Ruin and Recreate (Large Neighborhood Move): This is a powerful multistage move where the solver ruins part of the current solution and then reconstructs it, effectively performing a mini optimization within a move. In TimeFold’s implementation, the move will select a subset of entities and unassign them (stage 1), then immediately run a construction heuristic to reassign those entities (stage 2) before evaluating the new solution
docs.timefold.ai
docs.timefold.ai
. The result is a potentially very different neighbor solution, generated in one sweeping move. The ruin-and-recreate move is explicitly designed to help escape local optima: it lets the solver “undo a number of earlier decisions in one step, opening up a new part of the solution space”
docs.timefold.ai
. This is essentially an integrated form of large neighborhood search within TimeFold’s local search phase. Because it’s a costly, coarse-grained operation, it’s not used constantly, but rather occasionally (e.g. you can configure it to run with a low probability relative to simpler moves) so that the search sometimes makes a large jump
docs.timefold.ai

.
Composite Moves and Others: In general, any move that alters more than one variable or one entity can be seen as a multistage move. TimeFold supports composite moves like k-opt moves for vehicle routing (removing and reconnecting multiple route segments in one go), subchain moves in chained problems, and list moves that relocate or swap segments of a list. All these involve choosing multiple elements (e.g. multiple edges in a route, or two segments to swap) as part of constructing one move. The solver’s architecture treats move generation as a tree of selectors – for instance, it may first pick one part of the move, then conditionally pick the next part, and so on
docs.timefold.ai
. This staged selection is core to the multistage mechanism. Each composite move is evaluated by the scoring function as one neighbor solution, and thanks to incremental score calculation, evaluating a multi-change move is still efficient (the score delta is computed from the net effect of all the changes).
In summary, TimeFold’s multistage move mechanism expands the local search beyond simple one-step tweaks by incorporating moves composed of multiple changes or phases. These moves serve the purpose of navigating complex constraint landscapes – they can bypass transient constraint violations and address issues that require concerted adjustments. By fitting these into the normal local search loop (often mixed with simpler moves), TimeFold leverages both the fine-grained exploration and the coarse-grained leaps. The key principles include grouping related changes into one operation, using hierarchical selection to construct moves, and maintaining performance through techniques like just-in-time move generation and incremental scoring. This mechanism allows a practitioner familiar with optimization to trust that the solver isn’t just “hill-climbing” one small step at a time, but can also make intelligent multi-step jumps toward a better solution when needed

How it differs from union step?

Below is the **clean mental model** you want. I’ll be blunt and precise, because this distinction matters when you design search spaces.

---

## Short answer

**Union step** = *parallel choice of independent move types*
**Multistage move** = *one move composed of several dependent sub-steps*

They solve **different problems** in local search.

---

## Union step (Union Move Selector)

### What it is

A **union step** means:

> “At each local-search step, choose **one move** from *several alternative move generators*.”

Think **OR**, not AND.

Example:

* ChangeMove (single entity change)
* SwapMove (two-entity swap)
* PillarMove (group move)

A union selector says:

> *Try any of these move types; whichever scores best wins.*

### Properties

* Each move is **atomic**
* Each move is **complete by itself**
* No internal dependency between sub-choices
* Exploration = **breadth**

### Why it exists

Union steps are about **coverage**:

* Different move types explore different neighborhoods
* Prevents solver from being biased toward only one topology
* Ensures reachability of the whole solution space

### Mental image

```
Step N:
  ├─ try ChangeMove
  ├─ try SwapMove
  ├─ try PillarMove
  └─ pick best
```

Nothing inside those moves depends on anything else.

---

## Multistage move

### What it is

A **multistage move** is:

> “One logical move that is built through **several dependent decisions**, applied atomically.”

Think **AND**, with ordering.

Example (classic):

1. Select subset of entities
2. Remove them (ruin)
3. Reinsert them using a heuristic (recreate)
   → **single move**, single score evaluation

### Properties

* Internally **sequential**
* Later stages **depend on earlier ones**
* Intermediate states are **not scored**
* Final result is what matters
* Exploration = **depth / jump**

### Why it exists

Multistage moves exist to:

* Escape local minima that require **coordinated changes**
* Avoid invalid intermediate states
* Perform **large neighborhood search** efficiently
* Encode domain logic (routes, chains, workloads)

### Mental image

```
One Move:
  Stage 1 → Stage 2 → Stage 3
        ↓
   final solution evaluated
```

---

## Core difference (this is the key)

| Aspect        | Union step                 | Multistage move                 |
| ------------- | -------------------------- | ------------------------------- |
| Purpose       | Choose *which* move to try | Define *how* a move is built    |
| Composition   | OR (alternatives)          | AND (sequence)                  |
| Atomicity     | Each move atomic           | Whole sequence atomic           |
| Dependency    | None between moves         | Strong between stages           |
| Search effect | Wider exploration          | Larger jumps                    |
| Typical use   | Mix neighborhoods          | Escape traps / encode structure |

---

## Important insight (many people miss this)

> **Union is horizontal composition.
> Multistage is vertical composition.**

You often use **both at once**.

Example:

```
Union selector:
  ├─ Simple ChangeMove
  ├─ SwapMove
  └─ Ruin&Recreate (multistage move)
```

So:

* Union decides *which strategy*
* Multistage defines *how that strategy works internally*

---

## Practical rule of thumb (engineering mindset)

Use **union steps** when:

* You want diversity
* You’re unsure which neighborhood works best
* You need reachability guarantees

Use **multistage moves** when:

* Improvement requires multiple coordinated changes
* Single moves get stuck
* Problem has structure (routes, chains, batches, shifts)
* Temporary infeasibility must be avoided

---

## Why TimeFold emphasizes multistage moves

TimeFold (like OptaPlanner) targets:

* VRP
* Scheduling
* Chain-based planning
* Group constraints

These domains **cannot be solved well** with single-variable moves.

Multistage moves encode:

* “Remove → insert”
* “Select group → move together”
* “Break chain → reconnect”

Without multistage moves, the solver:

* Thrashes
* Plateaus early
* Needs extreme metaheuristic tuning

> **Union step chooses between different move types;
> multistage move defines a single move that requires several dependent decisions to exist at all.**
